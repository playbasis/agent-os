# Evidence Dashboard Harness Boundary

Last updated: 2026-07-06

The evidence dashboard is generated by the `pbos evidence-dashboard improve`
harness. Treat the checked-in report artifacts as output-only evidence, not as a
manual edit surface.

## Edit Surface

Hand-edit only these areas when changing the dashboard harness:

- `packages/cli/src/evidence-dashboard.ts` for generator, runner, prompt,
  source-data selection, snapshot, browser-check, and orchestration changes.
- CLI wiring, such as `packages/cli/src/index.ts` command routing/help text and
  command aliases, for exposing the harness.
- Documentation that describes the command, generated artifacts, and operating
  boundary.

Do not hand-edit these generated outputs:

- `reports/evidence-improvement-report.html`
- `reports/evidence-improvement-report-screenshot.png`
- `reports/evidence-improvement-report-mobile-screenshot.png`
- `reports/snapshots/*`

Those paths are regenerated by:

```bash
pnpm pbos evidence-dashboard improve --paths 20 --variants 6
```

Use `--paths`, `--variants`, or `--timestamp` only when the harness run needs
those explicit controls. `--skip-browser` is for debugging only; final promotion
must keep local browser checks enabled.

## Regeneration Rule

If the dashboard needs a content, layout, data, prompt, gate, screenshot, trace,
or snapshot change, change the harness first and rerun the command. The generated
HTML and snapshot bundle should be the result of the command, not a patch target.

Generated diffs are acceptable only when they are produced by the harness run and
the run's JSON result, mission report, improvement trace, screenshots, and
snapshot manifest explain the promoted artifact.

The final dashboard promotion must include the `generated-not-hand-edited` gate:
the checked-in dashboard HTML has to match the generator output byte-for-byte for
the promoted mission inputs.

## Visual Reference and Promotion

When live provider calls are enabled, the harness requests a GPT-image-2 visual
reference and stores the raw PNG under the private eval root outside this repo.
Public artifacts may record only safe metadata: status, hash, byte count,
dimensions, and whether the reference was available. Do not copy the private PNG
into reports, snapshots, commits, or public archives.

The command scores multiple design variants, records
`reports/evidence-dashboard-variant-scores-<timestamp>.json`, promotes the
highest measured candidate, then reruns render, text-fit, overflow, console,
visual-reference, leak, manifest, and archive gates against the promoted output.

## Review Checklist

Before finishing dashboard harness work:

- Confirm any intentional source changes are in `packages/cli/src/evidence-dashboard.ts`,
  CLI wiring, or docs.
- Confirm `reports/evidence-improvement-report.html` was not manually edited.
- Confirm every `reports/snapshots/*` change came from the dashboard harness
  command.
- Confirm the mission report and trace include a promoted variant, variant-score
  count, and `generated-not-hand-edited` pass gate.
- Confirm the GPT-image-2 reference, when available, is stored outside the repo
  and represented publicly only by safe metadata.
- Keep claim boundaries source-grounded: generated dashboard values must come
  from local report JSON or existing proof artifacts, and private payloads,
  provider secrets, held-out bodies, answer keys, and private screenshots must
  stay out of public artifacts.
